# 环境
1. 当前项目的python环境使用conda管理，建议使用conda创建一个新的环境来安装依赖。
2. 安装依赖：在项目根目录下运行以下命令安装所需
3. 当前项目的前端环境使用Node.js管理，建议使用nvm来管理Node.js版本。

# 背景
当前项目是为了实现一个`珠宝鉴定助手`的MVP程序, 它具备如下的能力:
1. 通过图片对一个珠宝进行粗粒度的鉴定
2. 对于鉴定结果进行结构化存储

# 资料
当前项目所有的开发资料和设计文档均放在`docs` 目录下，开发者可以在该目录下找到相关的资料和文档。

# 工作逻辑
1. 先指定计划, 对于有争议的点点, 或者不清晰的点, 经过Human-in-loop讨论清楚后开发.
2. 通过`docs`目录下的roadmap.md文件来查看当前项目的开发计划和进度。 如果当前没有roadmap.md文件, 可以通过查看`docs`目录下的其他文档来了解当前项目的开发计划和进度。
3. 在制定了`roadmap.md`文件后, 对于每个`路标`, 分拆成一个个的`任务`, 每个`任务`对应一个`issue`, 通过`issue`来跟踪每个`任务`的进度和状态。每个路标使用独立的文件来跟踪开发过程.
4. 每完成一个`任务`, 需要在`路标`对应的文件中记录下完成的内容和相关的资料
5. 每完成一个`路标`, 需要在`roadmap.md`文件中更新该路标的状态和相关的资料
6. 及时补充测试和执行git commit, 让每个commit包含一个完整的功能点, 以便于后续的代码回滚和问题排查.
7. 及时更新`docs`目录下的相关文档

# 代码规范

## 注释即文档(Code-as-Doc)

所有后端代码(Python / FastAPI)必须遵守"注释即文档"规范,使读者无需翻阅外部文档即可理解代码意图:

1. **`class` 必须有 docstring**:说明这个类的职责、不变量、典型用法。Pydantic 模型 / SQLAlchemy 模型也必须写,说明业务含义(不是字段名重复)。
2. **`def` / `async def` 必须有 docstring**:说明做什么、入参语义(非显然时)、返回值语义、可能抛出的业务异常。public API(被其它模块 import 的)是强制要求;私有 helper(`_xxx`)按需。
3. **关键业务逻辑必须有行内注释**:权限判定、字段裁剪、状态机迁移、跨云调用、配额扣减、加密 / 签名、SQL 性能敏感点等,写 `# 为什么这么写` 而不是 `# 这行做什么`。
4. **关键变量必须命名 + 注释**:涉及金额(单位)、时间(时区 / 单位)、ID(业务编号 vs DB 自增)、配额(剩余 vs 总额)、状态(枚举值含义)等容易踩坑的变量,要么名字自解释,要么紧邻一行注释说明语义与单位。
5. **TODO / FIXME / XXX 必须带 owner 与上下文**:`# TODO(Stage 2): wire AsyncSession from app/db/session.py` 而不是裸 `# TODO`。
6. **不写废话注释**:`# 创建一个 user`(下一行就是 `user = User()`)这种零信息量注释禁止;注释只负责"为什么"和"非显然的怎么样"。
7. **docstring 用中文**;变量 / 函数名用英文。中文 docstring 让 review 直接理解业务,英文标识符让代码可移植。
8. 前端代码(TypeScript / React)沿用同样精神,但更轻量:exported component / hook / util 必须有 JSDoc 一行说明用途;复杂 hook 必须说明副作用与依赖。
